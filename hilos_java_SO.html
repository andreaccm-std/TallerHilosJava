Hilos en Java

Sistemas Operacionales — Politécnico Grancolombiano

¿Qué es un hilo?

Un proceso es un programa en ejecución. Cuando abres Chrome, eso es un proceso. Un hilo (thread) es una tarea dentro de ese proceso. Chrome tiene un hilo para mostrar la interfaz, otro para descargar archivos, otro para reproducir audio — todos corren "al mismo tiempo" dentro del mismo programa.

Analogía — la cocina del restaurante

El restaurante es el proceso. Cada cocinero es un hilo. Todos comparten la misma cocina (memoria), los mismos ingredientes (variables) y los mismos utensilios (archivos). Trabajan en paralelo, pero si dos cocineros intentan usar el mismo cuchillo al mismo tiempo sin coordinarse, hay problema.

Proceso vs. hilo

AspectoProcesoHilo

¿Para qué se usan los hilos?

Concurrencia

Interfaz responsiva mientras se descarga un archivo en segundo plano.

Paralelismo

En CPUs multinúcleo, repartir trabajo pesado entre núcleos: comprimir imágenes, entrenar modelos.

Servidores

Un hilo por cliente. Un servidor web atiende miles de conexiones sin bloquear a los demás.

Eficiencia I/O

Mientras un hilo espera que el disco lea un archivo, otro hilo avanza con otro trabajo.

Lo que comparte y lo que no

COMPARTIDO entre hilos

  • Heap (objetos, instancias)
  • Variables de clase (static)
  • Archivos abiertos
  • Código del programa

PRIVADO de cada hilo

  • Stack (variables locales)
  • Contador de programa (PC)
  • Registros del procesador
  • Estado de excepción activa
La memoria compartida es la mayor ventaja de los hilos y su mayor riesgo. Dos hilos modificando la misma variable sin coordinación producen resultados incorrectos. A eso se le llama condición de carrera.

¿Qué es Runnable?

Runnable es una interfaz de Java que representa una tarea: algo que se puede ejecutar. Tiene un solo método obligatorio: run(), que contiene el código que el hilo ejecutará.

Analogía — la receta y el cocinero

Runnable es como una receta de cocina. Describe qué hay que hacer, pero no lo hace sola. Un cocinero (Thread) toma la receta y la ejecuta. Puedes darle la misma receta a varios cocineros para que trabajen en paralelo.

¿Por qué existe Runnable?

En Java, hay dos formas de decirle a un Thread qué debe hacer:

Forma 1: heredar de Thread y sobreescribir run() directamente.

Forma 2: crear un objeto Runnable con la tarea, y pasárselo al Thread como parámetro.

Java no tiene herencia múltiple. Si tu clase extiende Thread, ya no puede extender ninguna otra clase. Por eso se prefiere Runnable: separas la tarea del mecanismo de ejecución, y tu clase queda libre para extender lo que necesite.

Runnable como interfaz funcional

Como Runnable tiene un solo método (run()), Java lo reconoce como interfaz funcional. Eso permite escribirlo con una lambda, que es una forma corta de definir una función sin nombre:

// Forma larga: clase anónima que implementa Runnable explícitamente
Runnable tarea = new Runnable() {
    public void run() {
        System.out.println("Hola desde el hilo");
    }
};

// Forma corta: lambda — hace exactamente lo mismo con menos código
// () son los parámetros (run no recibe ninguno)
// -> separa parámetros del cuerpo
Runnable tarea = () -> System.out.println("Hola desde el hilo");

// Darle la tarea a un Thread y arrancarlo
Thread t = new Thread(tarea, "Hilo-1");
t.start();
La lambda () -> ... equivale a implementar el método run(). Los () son los parámetros (ninguno aquí) y lo que sigue de -> es el cuerpo del método.

¿Cuándo usar cada forma?

SituaciónRecomendación
Tarea simple, pocas líneasLambda directamente en el constructor del Thread
Tarea reutilizable o con estado internoClase que implementa Runnable
Necesitas acceder a métodos de Thread (prioridad, nombre)Clase que extiende Thread

Un hilo en Java pasa por 6 estados definidos en Thread.State. Conocerlos es clave para depurar problemas de concurrencia.

→

Los estados BLOCKED, WAITING y TIMED_WAITING son desvíos temporales desde RUNNING

EstadoQué significaCómo llega allí
NEWObjeto creado, hilo aún no iniciadonew Thread(...)
RUNNABLEListo para ejecutar; puede estar esperando CPU o usándolat.start()
BLOCKEDEsperando entrar a un bloque synchronizedOtro hilo tiene el monitor

Llamar t.start() no ejecuta run() inmediatamente. Pone el hilo en RUNNABLE y el scheduler del SO decide cuándo darle CPU. Pueden pasar milisegundos antes de que realmente corra.
Error frecuente: llamar t.run() directamente no crea ningún hilo nuevo. Ejecuta el método en el hilo actual como una llamada normal. El programa corre secuencial, no concurrente.

Dos formas de crear hilos en Java

Forma 1 — Extender Thread

// SinRunnable.java
public class SinRunnable {

    // Subclase de Thread: sobrescribimos run() con lo que queremos ejecutar
    static class MiThread extends Thread {
        public void run() {
            // getName() retorna el nombre asignado al hilo
            System.out.println("Hola desde run() en: " + getName());
        }
    }

    public static void main(String[] args) throws Exception {
        System.out.println("Hola desde main en: "
                           + Thread.currentThread().getName());

        Thread t = new MiThread();  // estado: NEW (creado, aún no iniciado)
        t.start();                  // crea el hilo del SO y ejecuta run() en él
        t.join();                   // main se bloquea hasta que t termine

        System.out.println("Fin main");
    }
}
Limitación: si MiThread extiende Thread, no puede extender ninguna otra clase. En proyectos reales esto es muy restrictivo.

Forma 2 — Implementar Runnable (recomendada)

// ConRunnable.java
public class ConRunnable {
    public static void main(String[] args) throws Exception {
        System.out.println("Hola desde main en: "
                           + Thread.currentThread().getName());

        // Lambda que implementa Runnable.run()
        // Thread.currentThread().getName() imprime el nombre del hilo que ejecuta esta línea
        Runnable tarea = () ->
            System.out.println("Hola desde run() en: "
                               + Thread.currentThread().getName());

        // Thread recibe la tarea y un nombre descriptivo
        Thread t = new Thread(tarea, "Hilo-1");
        t.start();  // arranca el hilo; run() se ejecutará en él
        t.join();   // main espera a que t termine antes de continuar

        System.out.println("Fin main");
    }
}

Reusar la misma tarea en varios hilos

// RunnableJoin.java
public class RunnableJoin {
    public static void main(String[] args) throws Exception {

        // Un solo Runnable reutilizado en dos hilos distintos
        // Cada hilo ejecuta el mismo código, pero de forma independiente
        Runnable tarea = () ->
            System.out.println("Trabajo en "
                               + Thread.currentThread().getName());

        Thread t1 = new Thread(tarea, "Trabajador-1");
        Thread t2 = new Thread(tarea, "Trabajador-2");

        t1.start();  // lanza t1
        t2.start();  // lanza t2 sin esperar a que t1 termine

        // join() = punto de sincronización: main espera a que ambos terminen
        t1.join();
        t2.join();

        // Esta línea solo se imprime cuando AMBOS hilos terminaron
        System.out.println("Listo: todos terminaron");
    }
}

sleep() y entrelazado de hilos

Analogía

Thread.sleep(300) es como un cocinero que dice "esperen 5 minutos mientras el agua hierve". Cede la CPU voluntariamente y el sistema puede usarla para avanzar con otros hilos.

// SleepYNombre.java
public class SleepYNombre {

    // Clase interna con estado propio (nombre del worker)
    static class Worker implements Runnable {
        private final String nombre;
        Worker(String nombre) { this.nombre = nombre; }

        public void run() {
            for (int i = 1; i <= 3; i++) {
                // Imprime: qué worker, qué paso, en qué hilo del SO corre
                System.out.println(nombre + " paso " + i
                                   + " en " + Thread.currentThread().getName());
                try {
                    Thread.sleep(300);  // cede CPU por 300ms (TIMED_WAITING)
                } catch (InterruptedException e) {
                    // Si interrumpen el hilo durante el sleep,
                    // restauramos la bandera en lugar de ignorarla
                    Thread.currentThread().interrupt();
                }
            }
        }
    }

    public static void main(String[] args) throws Exception {
        Thread a = new Thread(new Worker("A"), "Hilo-A");
        Thread b = new Thread(new Worker("B"), "Hilo-B");
        a.start(); b.start();
        a.join();  b.join();
        System.out.println("Terminaron A y B");
    }
}
El orden de salida de A y B varía en cada ejecución. No está garantizado porque el scheduler decide cuándo cada hilo retoma la CPU al despertar del sleep. Ejecuta el programa varias veces para comprobarlo.

Condición de carrera y sincronización

El problema: valor++ no es atómico

Una operación atómica es aquella que el procesador ejecuta de una sola vez, sin interrupciones. valor++ parece una operación, pero en realidad son tres pasos del bytecode Java:

1. Leer valor desde memoria → registro del CPU
2. Sumar 1 al valor en el registro
3. Escribir el registro de vuelta en memoria

Si dos hilos ejecutan esto simultáneamente, el segundo puede leer el valor viejo antes de que el primero lo haya escrito. Ambos leen 5, ambos calculan 6, ambos escriben 6. Se pierde un incremento.

Analogía — el pizarrón de ventas

Dos vendedores ven un pizarrón que dice "ventas: 50". Ambos van a sumar 1 al mismo tiempo. Ambos leen 50, ambos calculan 51, ambos escriben 51. El resultado debería ser 52, pero quedó en 51. Se perdió una venta.

// Carrera.java — demostración de condición de carrera
public class Carrera {

    static class Contador {
        int valor = 0;

        // PROBLEMA: valor++ no es atómica
        // Tres hilos pueden pisarse en los pasos leer/sumar/escribir
        void inc() {
            valor++;
        }
    }

    public static void main(String[] args) throws Exception {
        Contador c = new Contador();

        // La misma tarea: incrementar 100.000 veces
        Runnable tarea = () -> {
            for (int i = 0; i < 100_000; i++) c.inc();
        };

        // Tres hilos ejecutan la tarea simultáneamente sobre el mismo objeto
        Thread t1 = new Thread(tarea);
        Thread t2 = new Thread(tarea);
        Thread t3 = new Thread(tarea);

        t1.start(); t2.start(); t3.start();
        t1.join();  t2.join();  t3.join();

        // Debería imprimir 300.000, pero casi nunca lo hace
        System.out.println("Esperado=300000, Obtenido=" + c.valor);
    }
}
La condición de carrera no siempre produce el error. A veces el resultado es correcto por azar de timing. Eso la hace peligrosa: puede pasar a producción sin detectarse y fallar solo bajo carga alta.

Solución 1 — synchronized

El modificador synchronized en un método le pone un candado al objeto. Solo un hilo a la vez puede ejecutar métodos synchronized del mismo objeto. Los demás quedan BLOCKED hasta que el hilo actual termine y libere el candado.

Analogía — el baño con llave

El que entra cierra con llave. Los demás hacen fila afuera. Cuando sale, el siguiente puede entrar. Garantiza que nadie comparte el espacio, pero si hay mucha gente la fila crece.

Solución 2 — AtomicInteger

AtomicInteger usa instrucciones especiales del procesador (CAS: Compare-And-Swap) que son atómicas por hardware. No bloquea el hilo: si detecta conflicto, reintenta automáticamente. Es más eficiente que synchronized para contadores simples.

// ContadorSeguro.java — dos soluciones al mismo problema
import java.util.concurrent.atomic.AtomicInteger;

public class ContadorSeguro {

    // SOLUCIÓN 1: synchronized
    // El candado garantiza que solo un hilo ejecuta inc() o get() a la vez
    static class ContadorSync {
        private int valor = 0;

        // synchronized = candado sobre 'this' (este objeto)
        // Si t1 está en inc(), t2 queda BLOCKED hasta que t1 salga
        public synchronized void inc() { valor++; }
        public synchronized int get()  { return valor; }
    }

    // SOLUCIÓN 2: AtomicInteger
    // Usa instrucción CAS del procesador: atómica por hardware, sin bloquear hilos
    static class ContadorAtomic {
        AtomicInteger valor = new AtomicInteger(0);

        // incrementAndGet() = leer + sumar 1 + escribir, como una sola instrucción
        void inc() { valor.incrementAndGet(); }
        int get()  { return valor.get(); }
    }

    // Método auxiliar: ejecuta una prueba y mide cuánto tiempo tarda
    private static void correrPrueba(String nombre, Runnable test) {
        long t0 = System.currentTimeMillis();
        test.run();
        long t1 = System.currentTimeMillis();
        System.out.println(nombre + " tomó " + (t1 - t0) + " ms\n");
    }

    public static void main(String[] args) throws Exception {

        // Prueba 1: synchronized
        correrPrueba("ContadorSync", () -> {
            ContadorSync c = new ContadorSync();
            Runnable tarea = () -> { for (int i = 0; i < 100_000; i++) c.inc(); };
            Thread t1 = new Thread(tarea), t2 = new Thread(tarea), t3 = new Thread(tarea);
            try {
                t1.start(); t2.start(); t3.start();
                t1.join();  t2.join();  t3.join();
            } catch (InterruptedException ignored) {}
            System.out.println("Sync => " + c.get());   // siempre 300.000
        });

        // Prueba 2: AtomicInteger — mismo resultado, menor tiempo
        correrPrueba("ContadorAtomic", () -> {
            ContadorAtomic c = new ContadorAtomic();
            Runnable tarea = () -> { for (int i = 0; i < 100_000; i++) c.inc(); };
            Thread t1 = new Thread(tarea), t2 = new Thread(tarea), t3 = new Thread(tarea);
            try {
                t1.start(); t2.start(); t3.start();
                t1.join();  t2.join();  t3.join();
            } catch (InterruptedException ignored) {}
            System.out.println("Atomic => " + c.get()); // siempre 300.000
        });
    }
}

Solución 3 — Productor/consumidor con wait()/notifyAll()

A veces no basta con exclusión mutua: un hilo necesita esperar a que otro produzca un dato. Para eso existe la coordinación por condición.

Analogía — cocina y mesero

El cocinero (productor) prepara platos y los deja en la ventanilla. El mesero (consumidor) los recoge. Si la ventanilla está llena, el cocinero espera. Si está vacía, el mesero espera. Cuando uno termina, avisa al otro.

// Buffer1Elemento.java — patrón productor/consumidor
public class Buffer1Elemento {

    static class Buffer1 {
        private Integer dato = null;  // null = vacío, !null = lleno

        // El productor llama put() para dejar un dato
        public synchronized void put(int x) throws InterruptedException {
            // IMPORTANTE: usar while, no if
            // Al despertar del wait(), otro hilo pudo haber ocupado el espacio
            // El while re-verifica la condición antes de continuar
            while (dato != null) {
                wait();    // buffer lleno: libero el candado y me duermo
            }
            dato = x;      // deposito el dato en el buffer
            notifyAll();   // aviso a todos los que esperan (en especial al consumidor)
        }

        // El consumidor llama take() para retirar un dato
        public synchronized int take() throws InterruptedException {
            while (dato == null) {
                wait();    // buffer vacío: libero el candado y me duermo
            }
            int x = dato;  // tomo el dato
            dato = null;   // marco el buffer como vacío
            notifyAll();   // aviso al productor que hay espacio libre
            return x;
        }
    }

    public static void main(String[] args) throws Exception {
        Buffer1 buf = new Buffer1();

        // Productor: deposita los números 1..5 en el buffer
        Thread productor = new Thread(() -> {
            try {
                for (int i = 1; i <= 5; i++) {
                    buf.put(i);
                    System.out.println("Puso: " + i);
                }
            } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
        }, "Productor");

        // Consumidor: retira 5 elementos del buffer
        Thread consumidor = new Thread(() -> {
            try {
                for (int i = 1; i <= 5; i++) {
                    int x = buf.take();
                    System.out.println("Tomó: " + x);
                }
            } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
        }, "Consumidor");

        productor.start(); consumidor.start();
        productor.join();  consumidor.join();
        System.out.println("Fin");
    }
}
Patrón clave: siempre while con wait(), nunca if. Al despertar, la condición puede haber cambiado otra vez. El while la re-verifica antes de continuar.
